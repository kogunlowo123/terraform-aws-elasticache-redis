terraform-aws-elasticache-redis

ElastiCache Redis replication group with cluster mode, Multi-AZ, AUTH, encryption, and CloudWatch alarms
Terraform ≥ 1.0 AWS Provider ≥ 5.0 Redis 7.1 Encryption at Rest + In Transit

Executive Summary

This module deploys an Amazon ElastiCache Redis replication group with full support for cluster mode (sharding), Multi-AZ automatic failover, AUTH token authentication stored in AWS Secrets Manager, encryption at rest via KMS, encryption in transit via TLS, custom parameter groups, and four CloudWatch metric alarms (CPU, memory, evictions, connections). It provisions a dedicated security group, subnet group, and parameter group. The module supports both standard replication (primary + replicas) and cluster mode with configurable shards and replicas per shard.

Component Breakdown

Replication Group

  • Redis 7.1 engine (configurable)
  • Standard mode: num_cache_clusters (primary + replicas)
  • Cluster mode: num_node_groups x replicas_per_node_group
  • Automatic failover with Multi-AZ
  • Configurable backup window and retention
  • Maintenance window and auto minor version upgrades

Security

  • Dedicated VPC security group
  • Ingress from security group IDs or CIDR blocks
  • AUTH token (64-char password) stored in Secrets Manager
  • KMS encryption at rest (customer key or AWS managed)
  • TLS transit encryption enabled by default

Configuration

  • Custom parameter group (redis7 family)
  • User-defined parameters via custom_parameters map
  • Subnet group across multiple AZs
  • Log delivery to CloudWatch or Kinesis Firehose

Monitoring

  • CPU utilization alarm (threshold: 80%)
  • Database memory usage alarm (threshold: 80%)
  • Evictions alarm (threshold: > 0)
  • Current connections alarm (threshold: 50,000)
  • SNS notifications for alarm actions

Data Flow

  1. Application connects to Redis via primary or reader endpoint
  2. Security group validates inbound traffic on port 6379
  3. TLS handshake encrypts the connection in transit
  4. AUTH token validates client identity
  5. Write operations hit the primary node; reads can use replica endpoints
  6. Data at rest is encrypted via KMS key
  7. Automatic failover promotes a replica if primary fails (Multi-AZ)
  8. CloudWatch alarms monitor performance and notify via SNS

Industry Adaptation

E-Commerce / Session Cache

  • Standard mode with 2-3 replicas for read scaling
  • Low-latency cache.r6g instances
  • AUTH + TLS for PCI compliance
  • Eviction alarms prevent data loss

Real-Time Analytics

  • Cluster mode with multiple shards for throughput
  • High connection limit monitoring (50k threshold)
  • Log delivery to Kinesis Firehose

Healthcare / Financial

  • Customer-managed KMS keys
  • Transit encryption mandatory
  • Secrets Manager for credential rotation
  • SNS alarms for immediate incident response

Production Readiness Checklist

  • Enable at_rest_encryption_enabled and transit_encryption_enabled
  • Enable auth_token_enabled for authentication
  • Set automatic_failover_enabled = true and multi_az_enabled = true
  • Use at least 2 subnets across AZs
  • Configure snapshot_retention_limit ≥ 7
  • Provide notification_topic_arn for CloudWatch alarm actions
  • Use apply_immediately = false in production
  • Supply a customer-managed kms_key_arn for compliance
  • Review and tune custom_parameters for workload
  • Monitor all 4 CloudWatch alarms
